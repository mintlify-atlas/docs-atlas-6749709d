title: "Get started with OpenAgents Control"
description: "Install OAC, add your coding patterns, and run your first agent — from zero to an approved implementation plan in under five minutes."
---

OpenAgents Control runs on top of [OpenCode](https://opencode.ai). The installer sets up agents, commands, and context files in your project directory (or globally). Once installed, you start OpenCode with one of the OAC agents, add your patterns, and the approve-before-execute workflow handles the rest.

<Steps>
  <Step title="Check prerequisites">
    You need three things before installing:

    - **OpenCode CLI** — the open-source AI coding framework OAC builds on. The OAC installer will prompt you to set it up if it is not already installed. See the [OpenCode docs](https://opencode.ai/docs) for details.
    - **Bash 3.2 or higher** — macOS ships with Bash 3.2 by default; Linux distros typically ship with 4.0 or higher. Run `bash --version` to check.
    - **Git** — required for committing context files and sharing patterns with your team.

    <Note>
      On Windows, use Git Bash (recommended) or WSL. The installer does not support cmd.exe or PowerShell directly.
    </Note>
  </Step>

  <Step title="Install OAC">
    Run one command in your project directory. The `developer` profile installs the core agents, development subagents, commands, and context files needed for everyday coding work.

    <CodeGroup>

    ```bash developer (recommended)
    curl -fsSL https://raw.githubusercontent.com/darrenhinde/OpenAgentsControl/main/install.sh | bash -s developer
    ```

    ```bash essential (minimal)
    curl -fsSL https://raw.githubusercontent.com/darrenhinde/OpenAgentsControl/main/install.sh | bash -s essential
    ```

    ```bash full (everything except system-builder)
    curl -fsSL https://raw.githubusercontent.com/darrenhinde/OpenAgentsControl/main/install.sh | bash -s full
    ```

    </CodeGroup>

    This installs to `.opencode/` in your current directory. After installation your project structure will look like this:

    ```
    your-project/
    ├── .opencode/
    │   ├── agent/       # OpenAgent, OpenCoder, subagents
    │   ├── command/     # /add-context, /commit, /test, and more
    │   ├── context/     # MVI context files (core + project)
    │   └── tool/
    └── your-project-files...
    ```

    To install globally (available to all projects), add `--install-dir ~/.config/opencode`. See the [installation guide](/installation) for all options.
  </Step>

  <Step title="Add your patterns">
    This step is what makes OAC different from generic AI tools. Run `/add-context` inside OpenCode to teach agents your tech stack, conventions, and standards. It takes 10–15 minutes and answers 6 questions.

    ```bash
    opencode --agent OpenAgent
    ```

    Once inside OpenCode, run:

    ```
    /add-context
    ```

    You will be asked about:
    1. Your tech stack (for example: Next.js + TypeScript + PostgreSQL + Tailwind)
    2. An example API endpoint from your codebase
    3. An example component from your codebase
    4. Your naming conventions (kebab-case files, PascalCase components, camelCase variables)
    5. Your code standards (TypeScript strict mode, Zod validation, etc.)
    6. Your security requirements (input validation, parameterized queries, etc.)

    <Tip>
      You can skip this step and OAC will still work — but agents will generate generic code. Adding context is what produces output that matches your project from the first line.
    </Tip>

    After adding context, commit the files so your whole team benefits:

    ```bash
    git add .opencode/context/
    git commit -m "Add team coding standards to OAC context"
    git push
    ```
  </Step>

  <Step title="Run your first agent">
    Start OpenCoder for a production feature or OpenAgent for general tasks and questions.

    ```bash
    opencode --agent OpenCoder
    ```

    Then describe what you want to build:

    ```
    > "Create a user authentication system with JWT and refresh tokens"
    ```

    ContextScout immediately discovers relevant patterns from your context files — your validation approach, your ORM, your error format, your component structure.
  </Step>

  <Step title="Approve the plan">
    Before writing a single file, the agent presents a detailed implementation plan for your review. You approve or reject before any execution begins.

    ```
    ## Proposed implementation

    Components:
    - auth-guard.tsx (authentication wrapper)
    - login-form.tsx (login UI component)

    API endpoints:
    - /api/auth/login (POST)
    - /api/auth/refresh (POST)
    - /api/auth/session (GET)

    Database:
    - users table (Drizzle schema)
    - sessions table (Drizzle schema)

    All code will follow YOUR patterns from context.

    Approve? [y/n]
    ```

    Type `y` to proceed. The agent then implements incrementally — one component or endpoint at a time — with type checking and linting after each step.
  </Step>

  <Step title="What happens next">
    After you approve, execution follows a structured sequence:

    1. **Discover** (~1–2 min) — ContextScout loads your tech stack patterns, API conventions, component structure, and naming conventions
    2. **Execute** (~10–15 min) — Incremental implementation using your patterns, validated after each file
    3. **Validate** (~2–3 min) — TestEngineer writes tests, CodeReviewer checks for security issues
    4. **Ship** — Production-ready code that matches your project, ready to commit

    Total time for a complete feature with approval gates: roughly 15–25 minutes.

    <Note>
      Working with an external library for the first time? ExternalScout automatically fetches current documentation from official sources so agents use up-to-date API references, not outdated training data.
    </Note>
  </Step>
</Steps>

## Example workflow

Here is a concrete example showing what a real agent exchange looks like, from request to approved plan:

```text
$ opencode --agent OpenCoder
> "Add a Stripe webhook handler for subscription lifecycle events"

[ContextScout] Loading patterns...
  ✓ API pattern: Zod validation + Drizzle ORM
  ✓ Error handling: custom ApiError class
  ✓ Auth pattern: JWT middleware
  ✓ External: fetching Stripe webhook docs via ExternalScout

## Proposed implementation

Files to create:
- src/app/api/webhooks/stripe/route.ts
- src/lib/stripe/webhook-handlers.ts
- src/lib/stripe/subscription-sync.ts

Database changes:
- Add subscription_status column to users table (Drizzle migration)

All code follows your Zod + Drizzle + custom error patterns.

Approve? [y/n] y

[Implementing 1/3] src/app/api/webhooks/stripe/route.ts...
[Type check] ✓ Passed
[Implementing 2/3] src/lib/stripe/webhook-handlers.ts...
...
```
