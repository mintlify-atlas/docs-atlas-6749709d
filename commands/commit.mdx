The `/commit` command handles the full git commit workflow inside your OpenCode session. It analyzes your changes, determines the appropriate commit type and scope, generates a well-formatted message following the [Conventional Commits](https://www.conventionalcommits.org/) specification, and executes the commit after you confirm. There are two variants: the standard `/commit` for any project and `/commit-openagents` for work on the OAC repository itself.

## What `/commit` does

<Steps>
  <Step title="Pre-commit validation (optional)">
    Runs `pnpm lint` and `pnpm build` to catch issues before committing. If either fails, you can choose to fix the issues or proceed anyway. Pass a direct message as `$ARGUMENTS` to skip this step.
  </Step>
  <Step title="Analyze git status">
    Runs `git status --porcelain` to detect what has changed. If no files are staged, it automatically runs `git add .` to stage everything. If files are already staged, it commits only those.
  </Step>
  <Step title="Analyze the diff">
    Runs `git diff --cached` and reads the changes to determine the primary change type (`feat`, `fix`, `docs`, `refactor`, etc.) and the scope of the work.
  </Step>
  <Step title="Generate the commit message">
    Selects the matching emoji and type, then composes a message in the format `<emoji> <type>: <description>`. The description is written in imperative mood and kept under 72 characters.
  </Step>
  <Step title="Show for approval">
    Displays the proposed message so you can confirm or cancel before anything is committed.
  </Step>
  <Step title="Execute and push">
    Runs `git commit -m "<message>"` and then `git push`. Displays the commit hash and a brief summary on success.
  </Step>
</Steps>

## `/commit-openagents` вЂ” the OAC repository variant

`/commit-openagents` is a specialized variant designed for committing changes to the OAC repository itself. It differs from `/commit` in several ways:

<Tabs>
  <Tab title="Commit format">
    Uses conventional commits **without emojis**, with explicit scope paths:

    ```text
    feat(evals): add parallel test execution support
    fix(agents/core): correct delegation logic in openagent
    feat(subagents/code): add build validation to build-agent
    docs(readme): update installation instructions
    chore(deps): upgrade evaluation framework dependencies
    ```

    Valid scopes include `evals`, `agents/core`, `agents/meta`, `subagents/code`, `subagents/development`, `commands`, `context`, `scripts`, `ci`, `docs`, and `registry`.
  </Tab>
  <Tab title="Repo health check">
    Before staging anything, it runs a smart analysis:

    - Compares the `VERSION` file against the latest git tag and flags mismatches
    - Counts stale automation branches (`chore/version-bump-*`, `docs/auto-sync-*`) and suggests cleanup when there are more than three
    - Checks for disabled or missing CI workflows

    ```text
    рџ“Љ Repo Health Check:
    - Current branch: main
    - Version: 1.2.0 | Latest tag: v1.1.0
    - Stale branches: 2 automation branches
    - Uncommitted changes: 4 files

    вљ пёЏ Issues Found:
    - Missing release for v1.2.0 (tag not created)
    ```
  </Tab>
  <Tab title="Smart staging">
    Automatically stages only the files relevant to the detected change type вЂ” for example, `.opencode/agent/core/` for core agent changes or `.opencode/command/` for command changes. It never auto-stages `node_modules/`, `.env` files, or `evals/results/`.
  </Tab>
  <Tab title="Post-commit guidance">
    After pushing, explains which CI/CD workflows will be triggered and what to expect вЂ” for example, that a `feat:` commit to main will create a version-bump PR automatically.
  </Tab>
</Tabs>

## When to use each variant

| Situation | Command |
|---|---|
| Committing code in your own project | `/commit` |
| Committing changes to the OAC repository | `/commit-openagents` |
| Quick commit with a known message | `/commit "fix typo in README"` |

<Tip>
Both commands push automatically after committing. If you want to commit without pushing, you can cancel at the push step when prompted.
</Tip>
