OAC ships four commands focused on keeping your code correct, efficient, and consistent. They are independent of each other and can be run at any point in your workflow â€” before committing, after adding a feature, or as part of a periodic review.

## `/test` â€” run the full testing pipeline

The `/test` command runs your project's complete quality pipeline in a fixed sequence and repeats failed steps until everything passes.

**What it does:**

1. Runs `pnpm type:check` to catch TypeScript errors
2. Runs `pnpm lint` to enforce code style rules
3. Runs `pnpm test` to execute your test suite
4. Reports any failures with details
5. Applies fixes and repeats until all steps pass
6. Reports success

**Usage:**

```bash
/test
```

**When to use it:**

- Before running `/commit` to make sure you are not committing broken code
- After a large refactor to verify nothing broke
- When CI is failing and you want a local diagnosis

<Note>
`/test` expects `pnpm` scripts named `type:check`, `lint`, and `test` in your `package.json`. If your project uses a different package manager or script names, run the pipeline manually.
</Note>

## `/optimize` â€” analyze and improve code quality

The `/optimize` command takes a file path, a directory, or no argument at all and performs a multi-pass analysis covering performance, security, error handling, and maintainability. It produces a prioritized report with specific locations, impact assessments, and implementation guidance.

**Usage:**

```bash
/optimize                         # analyze recently changed files
/optimize src/api/users.ts        # analyze a specific file
/optimize src/components/         # analyze a directory
```

## `/validate-repo` â€” verify OAC repository integrity

`/validate-repo` is a read-only command that checks the entire OAC repository for internal consistency. It is primarily useful when you have added or modified agents, context files, or registry entries and want to confirm everything is properly wired together.

**Usage:**

```bash

```

**What it checks:**

## `/analyze-patterns` â€” find patterns and refactoring opportunities

`/analyze-patterns` scans your codebase for recurring code patterns, groups similar implementations by similarity score, and suggests where to consolidate or standardize.

**Usage:**

```bash
/analyze-patterns                                        # scan current directory with default depth
/analyze-patterns --pattern="error-handling" --language=ts
/analyze-patterns --pattern="factory" --depth=deep --output=json
/analyze-patterns --pattern="api-endpoint" --language=js --output=markdown
```

**Parameters:**

| Parameter | Description |
|---|---|
| `--pattern` | Pattern name or regex to search for. Predefined names include `singleton`, `factory`, `observer`, `error-handling`, `async-patterns`, `api-endpoint`, `middleware`. |
| `--language` | Filter by language: `js`, `ts`, `py`, `go`, `rust`, `java`, etc. |
| `--depth` | Search depth: `shallow` (current directory), `medium` (`src/`), or `deep` (entire repository). |
| `--output` | Output format: `text` (default), `json`, or `markdown`. |

**Example output (text format):**

```text
Pattern Analysis Report
=======================

Pattern: error-handling
Occurrences: 12
Files: src/api/users.ts, src/api/orders.ts, src/lib/db.ts

Implementations:
  1. src/api/users.ts:42  - try-catch with Response.json error (similarity: 98%)
  2. src/api/orders.ts:67 - try-catch with Response.json error (similarity: 95%)
  3. src/lib/db.ts:18     - try-catch with console.error only  (similarity: 72%)

Refactoring Suggestions:
  - Extract shared error handler to src/lib/handle-error.ts
  - Standardize db.ts to use the same Response.json pattern
  - Consider a higher-order function to wrap route handlers

  - 10 of 12 implementations are consistent (83%)
  - 2 implementations deviate: consider updating to match majority
```

**When to use it:**

- Before a refactoring session to understand how widespread a pattern is
- When onboarding onto a new codebase to learn its conventions
- When you suspect code duplication but want a concrete map before cleaning up
- To generate a markdown report to share with your team

<Tip>
Use `/analyze-patterns` before running `/add-context` on an existing project â€” the pattern report gives you concrete examples to paste into the wizard's API and component questions.
</Tip>
