The context system stores your coding standards as markdown files inside `.opencode/context/`. Every directory has a specific purpose, and understanding that structure helps you place new files correctly and keep your context lean enough for fast, cheap agent loading.

## Category reference

<AccordionGroup>
  <Accordion title="core/ — universal standards" defaultOpen>
    Standards and workflows that apply to every project and every language. These files ship with OAC and are loaded by agents regardless of your tech stack.

    | Subdirectory | What it contains |
    |---|---|
    | `standards/` | Code quality, security, testing, documentation, code analysis |
    | `workflows/` | Code review, task delegation, feature breakdown, session management, design iteration |
    | `guides/` | How-to guides for working within the context system (e.g., resuming sessions) |
    | `task-management/` | JSON-driven task tracking schema and CLI command lookups |

    Files here are organized by function: `context-system.md` describes the system itself, `essential-patterns.md` covers the most-used patterns across all categories.
  </Accordion>

  <Accordion title="project-intelligence/ — project-specific patterns">
    Your project's unique coding patterns, tech stack decisions, and business context. This is the most important category — project intelligence is **always local only** and is never loaded from a global installation.

    | File | What it captures |
    |---|---|
    | `technical-domain.md` | Tech stack, architecture pattern, key technical decisions |
    | `business-domain.md` | Problem statement, target users, value proposition |
    | `business-tech-bridge.md` | How business needs map to technical solutions |
    | `decisions-log.md` | Major decisions with rationale and alternatives considered |
    | `living-notes.md` | Active issues, technical debt, open questions |

    Created and updated via `/add-context`. See [project intelligence](/reference/project-intelligence) for the full workflow.
  </Accordion>

  <Accordion title="ui/ — design system and component patterns">
    Frontend and design patterns. The `web/` subdirectory contains standards for Tailwind CSS, Flowbite components, React hooks, animation micro-interactions, and design system principles.

    Organized by concern, then technology:
    - `ui/web/react-patterns.md` — modern React hooks, component composition, state management
    - `ui/web/ui-styling-standards.md` — Tailwind CSS conventions, responsive design
    - `ui/web/animation-patterns.md` — micro-interactions and timing standards
    - `ui/web/design-systems.md` — design system selection and principles
  </Accordion>

  <Accordion title="development/ — language and backend patterns">
    Development-specific context organized by concern (frontend, backend, data) and then by technology. Uses the concern-based directory pattern.

    Specialized navigation files provide cross-cutting routes:
    - `development/backend-navigation.md` — APIs, auth, middleware, Node.js
    - `development/ui-navigation.md` — spans `frontend/` and `ui/`

    Add new technologies at the appropriate level: full-stack frameworks under `development/frameworks/{tech}/`, layer-specific technologies under `development/{frontend|backend}/{tech}/`.
  </Accordion>

  <Accordion title="content-creation/ — content-focused context">
    Context for agents working on documentation, copywriting, and content workflows. Add content style guides, tone-of-voice standards, and content patterns here.
  </Accordion>

  <Accordion title="openagents-repo/ — OAC's own meta-context">
    Context about the OpenAgentsControl repository itself. Uses the function-based directory pattern with `concepts/`, `examples/`, `guides/`, `lookup/`, and `errors/` subdirectories. Only relevant when developing OAC itself.
  </Accordion>
</AccordionGroup>

## File naming conventions

All context files use **kebab-case** names that describe their contents without needing to open the file.

<CardGroup cols={2}>
  <Card title="Good names" icon="check">
    `code-quality.md`  
    `security-patterns.md`  
    `react-patterns.md`  
    `resuming-sessions.md`
  </Card>
  <Card title="Avoid" icon="x">
    `code.md`  
    `security.md`  
    `react.md`  
    `sessions.md`
  </Card>
</CardGroup>

Specialized navigation files that span multiple categories use a domain prefix: `backend-navigation.md`, `ui-navigation.md`, `fullstack-navigation.md`.

## File size limits

OAC uses the **Minimal Viable Information (MVI)** principle. Files are designed to be scannable in under 30 seconds.

| File type | Line limit | Purpose |
|---|---|---|
| Concept files | < 100 lines | What something is — core ideas only |
| Guide files | < 150 lines | How to do something — step-by-step workflows |
| Example files | < 80 lines | Working code patterns with minimal explanation |
| Navigation files | ~200–300 tokens | ASCII tree + quick-routes table + by-type index |

<Warning>
Files exceeding 200 lines accumulate token overhead on every agent invocation. If a file grows beyond this limit, split it by concern or apply `/context compact {file}` to reduce it to MVI format.
</Warning>

## What goes in each file type

<Tabs>
  <Tab title="Concept files">
    Extract only:
    - Core definition (1–3 sentences)
    - Key properties (3–5 bullet points)
    - One minimal working example
    - Link to full reference documentation

    Do not include: verbose explanations, complete API docs, marketing content, or duplicate information from other files.
  </Tab>
  <Tab title="Guide files">
    Include:
    - Numbered step-by-step workflow
    - Decision points and conditionals
    - Expected outputs per step
    - Common errors and how to resolve them

    Keep each step to 1–3 sentences. Link to concept files for background rather than repeating it.
  </Tab>
  <Tab title="Example files">
    Include:
    - A single, complete, working code pattern
    - Inline comments on non-obvious lines only
    - The specific scenario this example addresses

    Do not pad with setup code, boilerplate, or alternative approaches. One example per file.
  </Tab>
  <Tab title="Navigation files">
    Every category and subcategory requires a `navigation.md`. It must contain:
    - An ASCII directory tree (~50 tokens)
    - A quick-routes table mapping tasks to file paths (~100 tokens)
    - A "by type" or "by concern" section (~50 tokens)

    Navigation files are the map ContextScout uses to discover relevant files without reading every file in a category.
  </Tab>
</Tabs>

## Real example: a context file

The following is the actual `project-intelligence/navigation.md` from the OAC repository, showing the recommended structure for a navigation file:

```markdown project-intelligence/navigation.md
<!-- Context: project-intelligence/nav | Priority: high | Version: 1.0 | Updated: 2025-01-12 -->

# Project Intelligence

> Start here for quick project understanding. These files bridge business and technical domains.

## Structure

.opencode/context/project-intelligence/
├── navigation.md              # This file - quick overview
├── business-domain.md         # Business context and problem statement
├── technical-domain.md        # Stack, architecture, technical decisions
├── business-tech-bridge.md    # How business needs map to solutions
├── decisions-log.md           # Major decisions with rationale
└── living-notes.md            # Active issues, debt, open questions

## Quick Routes

| What You Need | File | Description |
|---|---|---|
| Understand the "why" | business-domain.md | Problem, users, value proposition |
| Understand the "how" | technical-domain.md | Stack, architecture, integrations |
| See the connection | business-tech-bridge.md | Business → technical mapping |
| Know the context | decisions-log.md | Why decisions were made |
| Current state | living-notes.md | Active issues and open questions |
```

<Tip>
Every new context category you create — including subfolders inside `project-intelligence/` — must start with a `navigation.md`. ContextScout reads these files to navigate the tree without loading every file.
</Tip>
