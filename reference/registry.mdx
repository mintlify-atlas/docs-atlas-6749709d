The `registry.json` file at the root of the OpenAgents Control repository is the authoritative manifest for every installable component. The installer reads this file at runtime to resolve profiles, validate dependencies, and locate source files. Understanding its structure lets you add custom components, audit dependency chains, and verify registry integrity with the built-in validation script.

## Schema and top-level structure

The registry is a single JSON object at schema version `2.0.0`. Its top-level keys are:

```json
{
  "version": "2.0.0",
  "schema_version": "2.0.0",
  "repository": "https://github.com/darrenhinde/OpenAgentsControl",
  "categories": { ... },
  "components": { ... },
  "profiles": { ... },
  "metadata": { ... }
}
```

The `metadata` object records when the registry was last updated and the schema version used:

```json
"metadata": {
  "lastUpdated": "2026-03-21",
  "schemaVersion": "1.0.0"
}
```

## Component categories

Every component belongs to exactly one category. The five canonical categories are:

| Category | Purpose |
|---|---|
| `essential` | Minimal components required for basic functionality |
| `standard` | Standard components for typical use |
| `extended` | Extended components for advanced features |
| `specialized` | Specialized components for specific domains |
| `meta` | Meta-level components for system generation |

Additional non-canonical categories appear in the registry for organisational purposes: `core`, `content`, `data`, `development`, `utils`, `test`, and `navigation`.

## Component types

The `components` object contains arrays keyed by component type. The registry defines eight types: `agents`, `subagents`, `commands`, `tools`, `plugins`, `skills`, `contexts`, and `config`.

### Common component schema

Every component entry shares the same base shape:

```json
{
  "id": "unique-id",
  "name": "Human Readable Name",
  "type": "agent | subagent | command | tool | plugin | skill | context | config",
  "path": ".opencode/relative/path/to/file.md",
  "description": "What this component does",
  "tags": ["tag1", "tag2"],
  "dependencies": ["type:id", "type:id"],
  "category": "essential | standard | ..."
}
```

Dependencies are expressed as `type:id` strings, for example `"subagent:contextscout"` or `"context:standards-code"`. The installer and the `validate:registry` script use these strings to detect missing components and circular references.

The registry lists over 100 context files in the `contexts` array. They span categories including core standards, workflows, system-builder templates, UI patterns, development principles, project intelligence, and navigation guides. Notable examples:

## Declaring dependencies

A component declares its dependencies as an array of `type:id` strings:

```json
{
  "id": "opencoder",
  "dependencies": [
    "subagent:documentation",
    "subagent:coder-agent",
    "subagent:tester",
    "subagent:reviewer",
    "subagent:build-agent",
    "subagent:contextscout",
    "subagent:externalscout",
    "context:standards-code",
    "context:task-delegation-basics",
    "context:component-planning",
    "context:external-libraries-workflow"
  ]
}
```

Glob patterns are also supported in profile component lists (e.g., `"context:core/*"` installs every context under the `core/` path prefix).

## Adding custom components

Follow the path conventions already used in the registry:

```
.opencode/agent/core/my-agent.md          # Orchestrator agents
.opencode/agent/subagents/code/my-sub.md  # Code-focused subagents
.opencode/command/my-command.md           # Slash commands
.opencode/tool/my-tool/index.ts           # TypeScript tools
.opencode/skills/my-skill/SKILL.md        # Skills
.opencode/context/my-domain/my-file.md    # Context files
```

Then add a matching entry to `registry.json` under the correct type array.

<Note>
  Custom components added to a local `registry.json` are only visible to that installation. They are not merged with the upstream registry when you run `./install.sh` again.
</Note>

## Validating registry integrity

The `validate:registry` npm script cross-checks every component entry against the file system and validates that all declared dependencies resolve:

```bash
# Standard validation
npm run validate:registry

# Verbose output
npm run validate:registry:verbose

# Attempt to auto-fix minor issues
npm run validate:registry:fix
```

<Tip>
  Run `npm run validate:registry` before committing changes to `registry.json` to catch missing files or broken dependency references before they reach CI.
</Tip>
