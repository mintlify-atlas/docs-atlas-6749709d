OpenCoder — OAC's development specialist agent — is designed to adapt to the language of the project it is working in. Language detection is automatic: OpenCoder reads project files such as `package.json`, `requirements.txt`, `go.mod`, `Cargo.toml`, or `*.csproj` to identify the stack, then switches to the appropriate runtime, type-checker, linter, and test runner for that language. You do not need to configure language support; it is inferred from what already exists in the project.

### TypeScript / JavaScript

**Language-specific conventions OpenCoder follows:**

- Functional, type-first approach
- Explicit types over inference where clarity is improved
- Immutable data structures and pure functions where possible
- Type guards and null safety throughout
- Strong preference for `async`/`await` over callbacks

### Rust

<CardGroup cols={2}>
  <Card title="Build" icon="hammer">
    `cargo check` and `cargo build` — detected by `Cargo.toml`.
  </Card>
  <Card title="Linting" icon="magnifying-glass-chart">
    `cargo clippy` — Rust's official linter with pedantic suggestions enabled.
  </Card>
  <Card title="Testing" icon="vial">
    `cargo test` — unit tests inline with `#[cfg(test)]` and integration tests under `tests/`.

**Language-specific conventions:**

- Ownership and borrowing handled explicitly — no unnecessary `clone`
- `Result` and `Option` used consistently; `.unwrap()` only in tests
- Traits preferred over dynamic dispatch where performance matters
- Error types implement `std::error::Error` or use `thiserror`

**Language-specific conventions:**

- Minimal APIs for web endpoints (ASP.NET Core)
- CQRS pattern with MediatR for command/query separation
- Records for immutable data transfer objects
- `async`/`await` throughout with `CancellationToken` support
- Nullable reference types enabled; no `!` (null-forgiving) operators without justification

### Other languages

OpenCoder's context system is language-agnostic. For languages not listed above, OpenCoder adapts by:

- Reading existing project files to understand conventions already in use
- Following idiomatic patterns for the language detected
- Running whatever build and test commands are configured in the project (e.g., `Makefile` targets)
- Delegating to ContextScout to find any internal context files documenting language-specific standards

<Info>
  The context system stores language and framework-specific guides in `.opencode/context/development/`. If your project uses a language that benefits from custom conventions, create a context file there and ContextScout will surface it automatically.
</Info>

## Language detection workflow

OpenCoder reads the project root for well-known configuration files in this order:

1. `package.json` / `bun.lock` / `deno.json` → TypeScript/JavaScript
2. `requirements.txt` / `pyproject.toml` / `setup.py` → Python
3. `go.mod` → Go
4. `Cargo.toml` → Rust
5. `*.csproj` / `*.sln` → C#/.NET

When multiple languages are present (e.g., a Python backend with a TypeScript frontend), OpenCoder detects both and applies the correct toolchain to each subdirectory independently.

## Validation cadence

Regardless of language, OpenCoder validates incrementally — not just at the end of implementation:

```
For each implementation step:
  1. Write the code
  2. Run type checks (tsc / mypy / Roslyn / cargo check)
  3. Run linting (eslint / pylint / golangci-lint / clippy / dotnet format)
  4. Run relevant tests
  5. Confirm no regressions before moving to the next step
```

This cadence catches errors early, keeps the codebase in a working state throughout, and makes rollbacks straightforward.

## Compatibility notes

The install script and agent framework itself require only bash 3.2+, `curl`, and `jq`. No language runtime is needed to install OAC. Language runtimes are only required at implementation time when OpenCoder invokes type checkers and test runners in your project.

| Platform | Bash version | Status |
|---|---|---|
| macOS 13+ (default bash) | 3.2.57 | Fully supported |
| macOS 13+ (Homebrew bash) | 5.2 | Fully supported |
| Ubuntu 24.04 | 5.2.21 | Fully supported (fixed in v1.1.0) |
| Ubuntu 22.04 | 5.1.16 | Fully supported |
| Ubuntu 20.04 | 5.0.17 | Fully supported |
| Debian 11 | 5.1.4 | Fully supported |
| Fedora 38 | 5.2.15 | Fully supported |
| Arch Linux | 5.2.21 | Fully supported |
| Alpine 3.18 | 5.2.15 | Fully supported (requires bash package) |
| Windows — Git Bash | 4.4.23 | Fully supported |
| Windows — WSL2 Ubuntu | 5.1.16 | Fully supported |
| Windows — cmd.exe | N/A | Not supported |
| Bash < 3.2 | — | Not supported |

<Note>
  A compatibility issue with bash 5.x on Ubuntu 24.04 (where `set -e` triggered premature exits on arithmetic increment operations) was fixed in v1.1.0 by changing all `((variable++))` expressions to `variable=$((variable + 1))`. If you encounter unexpected exits on modern Linux, update to the latest installer.
</Note>
