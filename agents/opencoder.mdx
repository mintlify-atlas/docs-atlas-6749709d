OpenCoder is the development specialist in OpenAgents Control. Where OpenAgent handles general tasks and simple code changes, OpenCoder is built specifically for the kind of work that requires careful planning, incremental execution, and thorough validation — complex features spanning multiple files, deep architecture refactoring, pattern implementation across a codebase, and multi-language implementations. It orchestrates a team of subagents, executes tasks in parallel where dependencies allow, and validates each step before proceeding to the next.

## Invoking OpenCoder

```bash
opencode --agent OpenCoder
```

## When to use OpenCoder

<CardGroup cols={2}>
  <Card title="Multi-file refactoring" icon="arrows-rotate">
    Refactoring a module, implementing a pattern across multiple components, or restructuring architecture that touches 4 or more files.
  </Card>
  <Card title="Architecture analysis" icon="diagram-project">
    Analyzing codebase structure, identifying architectural issues, proposing and implementing design improvements.
  </Card>
  <Card title="Complex implementations" icon="layer-group">
    Features spanning multiple modules, implementations requiring more than 60 minutes, or work with complex inter-component dependencies.
  </Card>
  <Card title="Deep codebase analysis" icon="magnifying-glass">
    Understanding complex code flows, documenting architecture, identifying technical debt and proposing systematic reduction.
  </Card>
</CardGroup>

Use **OpenAgent** instead for questions, documentation tasks, simple 1–3 file changes, and general workflow coordination.

## The 3-phase workflow

<Steps>
  <Step title="Planning (required)">
    OpenCoder starts by calling ContextScout to discover relevant project context files, and ExternalScout for any external packages involved. It then presents a lightweight proposal for your approval — not a full plan document, just a summary of what it intends to build and how.

    ```
    ## Proposed Approach

    What: JWT-based authentication with refresh tokens and RBAC
    Components: Auth service, JWT utility, RBAC middleware, user model
    Approach: Delegate to TaskManager for breakdown (4+ files, >60 min)
    Context discovered: .opencode/context/core/standards/code-quality.md
    External docs: Fetch jsonwebtoken, bcrypt docs via ExternalScout

    Approval needed before proceeding.
    ```

    No session directory, no task files, and no code is created until you approve. If you reject or redirect, OpenCoder returns to the discovery stage.
  </Step>
  <Step title="Implementation (after approval)">
    After approval, OpenCoder initializes a session, creates a `context.md` file that becomes the single source of truth for all downstream agents, and delegates to TaskManager for complex work (4+ files, >60 minutes). Tasks are then executed in parallel batches based on their dependency graph.

    For each step in the plan:

    1. Implement the code
    2. Run type checks (if applicable)
    3. Run linting (if configured)
    4. Run build checks
    5. Execute relevant tests
    6. Validate results
    7. Move to the next step

    Validation happens continuously, not just at the end. Parallel tasks (those with no inter-dependencies) are dispatched simultaneously to CoderAgent instances, while sequential tasks wait for their dependencies to complete.
  </Step>
  <Step title="Completion">
    When implementation is complete, OpenCoder runs a final full-system validation and then recommends handoff to the appropriate specialists based on what the work needs:

    - `@tester` — if tests have not yet been written
    - `@reviewer` — if a security or quality review is needed
    - `@documentation` — if API docs or guides need generating
  </Step>
</Steps>

## Multi-language support

OpenCoder detects the project language from the files it encounters and adapts its toolchain automatically.

| Language | Type checking | Linting | Testing |
|---|---|---|---|
| **TypeScript / JavaScript** | `tsc` | `eslint` | `jest`, `vitest`, `mocha` |
| **Python** | `mypy` | `pylint`, `flake8` | `pytest`, `unittest` |
| **Go** | `go build` | `golangci-lint` | `go test` |
| **Rust** | `cargo check`, `cargo build` | `clippy` | `cargo test` |
| **C# / .NET** | Roslyn compiler (built-in) | `dotnet format`, Roslyn analyzers | `dotnet test` (xUnit + Shouldly + NSubstitute) |

## Code standards

OpenCoder applies consistent standards across all languages it works in.

<AccordionGroup>
  <Accordion title="Modular architecture">
    Clear module boundaries with single responsibility. Loose coupling and high cohesion. Dependency injection where appropriate. Each component owns its concerns and exposes clean interfaces.
  </Accordion>
  <Accordion title="Functional patterns">
    Pure functions where possible. Immutable data structures. Declarative over imperative patterns. Function composition over deep class hierarchies.
  </Accordion>
  <Accordion title="Type safety">
    Strong typing when the language supports it. Explicit types over inference when they add clarity. Type guards and validation at boundaries. Null safety enforced throughout.
  </Accordion>
  <Accordion title="Clean code">
    Meaningful names at every level. Small, focused functions that do one thing. Minimal, high-signal comments only. No over-complication. Language conventions respected (idiomatic Go, Pythonic Python, etc.).
  </Accordion>
</AccordionGroup>

## Subagent delegation

OpenCoder coordinates a team of specialized subagents. You do not invoke these manually — OpenCoder determines when each is needed.

| Subagent | When invoked | Output |
|---|---|---|
| **TaskManager** | Features spanning 4+ files or >60 minutes | Atomic JSON subtasks under `.tmp/tasks/{feature}/` |
| **CoderAgent** | Simple, focused implementation subtasks | Implemented code following specifications |
| **TestEngineer** | Tests needed after implementation | Unit, integration, and e2e test suites |
| **CodeReviewer** | Security or quality review needed | Review report with recommendations |
| **BuildAgent** | Build validation needed | Build status and error reports |
| **DocWriter** | Comprehensive documentation needed | API docs, guides, structured documentation |

## Examples

<Tabs>
  <Tab title="Multi-file refactoring">
    ```bash
    opencode --agent OpenCoder
    > "Refactor the authentication module to use dependency injection across all 8 files"

    # OpenCoder will:
    # 1. Analyze current structure (8 files)
    # 2. Propose refactoring plan
    # 3. Wait for approval
    # 4. Delegate to TaskManager (8 files exceeds 4-file threshold)
    # 5. Implement subtasks in parallel where possible
    # 6. Validate incrementally after each step
    # 7. Complete when all subtasks done
    ```
  </Tab>
  <Tab title="Complex feature">
    ```bash
    opencode --agent OpenCoder
    > "Implement user authentication with JWT, refresh tokens, and role-based access control"

    # OpenCoder will:
    # 1. Analyze requirements (complex, multi-file)
    # 2. Design authentication architecture
    # 3. Propose implementation plan (multiple phases)
    # 4. Wait for approval
    # 5. Delegate to TaskManager (create subtasks)
    # 6. Implement Phase 1: JWT infrastructure
    # 7. Implement Phase 2: Refresh token mechanism
    # 8. Implement Phase 3: RBAC system
    # 9. Coordinate with TestEngineer for test coverage
    # 10. Coordinate with CodeReviewer for security review
    ```
  </Tab>
  <Tab title="Architecture analysis">
    ```bash
    opencode --agent OpenCoder
    > "Analyze the architecture of this codebase and suggest improvements"

    # OpenCoder will:
    # 1. Scan codebase structure
    # 2. Identify patterns and anti-patterns
    # 3. Propose architectural improvements
    # 4. Wait for approval
    # 5. Implement approved changes
    # 6. Validate with build and tests
    ```
  </Tab>
</Tabs>

## Configuration

OpenCoder is configured in `.opencode/agent/core/opencoder.md` (local install) or `~/.config/opencode/agent/core/opencoder.md` (global install).

Default settings:

```yaml
---
name: OpenCoder
description: "Orchestration agent for complex coding, architecture, and multi-file refactoring"
temperature: 0.1
---
```

The `temperature: 0.1` setting makes OpenCoder deterministic and precise — the right default for production code generation.

To override the model:

```yaml
---
model: anthropic/claude-sonnet-4-5
---
```

<Note>
By default, all agents use your OpenCode default model. Configure a model override per agent only when you want different agents to use different models — for example, a cheaper model for simple subagents and a more capable model for the orchestrator.
</Note>
